Blueprint Latest Tech - Liquid Clustering

Why NOT during historical bulk load

  • 650+ Gold tables had to mirror Hadoop for cutover validation
  • 2+ PB one-time load - inherited partitioning, proven write paths
  • Clustering keys need day-forward access patterns, not bulk backfill
  • Physical layout change mid-migration added risk with no incremental benefit

Why AFTER 6/1/2026 (implemented)

  • Incremental Bronze to Silver to Gold live - ongoing writes and queries
  • Liquid Clustering replaced regular partitioning on Gold
  • Better fit for 650+ tables (0.0001 GB - 150 TB) and evolving filters
  • Post-cutover optimization - latest Delta features when workloads were real

Panel line: staged latest tech deliberately - mirror first, optimize once incremental workloads were real.

Blueprint MDFSG Components

  • framework_process_config - source, target, SCD, SQL transforms
  • MDFSG UI - developers configure loads without custom notebooks
  • framework_main - reusable notebook executes metadata-driven writes
  • silver/gold_version_history - CDF versioning for incremental reads
